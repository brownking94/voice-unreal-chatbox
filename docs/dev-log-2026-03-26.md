# Development Log — March 26, 2026

Session covering CUDA GPU acceleration, multilingual STT, broadcast architecture, and Unreal Engine integration.

## What Was Built

Starting from a working macOS prototype (whisper.cpp + test client over TCP), this session ported everything to Windows with NVIDIA CUDA GPU acceleration, added multilingual speech-to-text, broadcast messaging, and documented the Unreal Engine integration.

## Key Technical Decisions

### CUDA GPU Acceleration

- **whisper.cpp v1.7.4 → v1.8.4**: v1.7.4 had a known bug (`"TODO: CUDA is currently broken"`) where the CUDA backend loaded but inference silently fell back to CPU. This caused 60-second transcription times for 30-second audio clips. v1.8.4 fixes this.
- **sm_89 instead of sm_120a**: The RTX 5070 is a consumer Blackwell GPU (sm_120). ggml's CMake forcibly rewrites `120` → `120a` to enable MXFP4 block-scale MMA instructions. These only work on data-center Blackwell (B200), not consumer cards. Targeting sm_89 (Ada Lovelace) sidesteps this — it's forward-compatible with all RTX 30/40/50 series.
- **Driver version matters**: CUDA Toolkit 13.2 requires a driver that supports CUDA 13.2+. The original driver (577.05) only supported CUDA 12.9, causing silent fallback to CPU with 0 MiB GPU usage. Updating to driver 595.97 fixed it.
- **RelWithDebInfo, not Debug**: Debug mode (`/Od`) made whisper inference extremely slow due to no compiler optimization. RelWithDebInfo gives full `/O2` optimization with debug symbols.

### Build System

- **Incremental builds are critical**: CUDA kernel compilation takes 10-15 minutes. Never delete the build directory. With MSVC multi-config generators, Debug/Release/RelWithDebInfo coexist in the same build directory.
- **First build warning**: Document that the first build is slow so people don't think it's broken.

### VAD Tuning

- **Threshold raised from 0.005 to 0.02**: The Windows mic had a higher noise floor than the Mac mic. At 0.005, background noise kept the VAD in "speech" state indefinitely, causing it to always hit the max chunk cap.
- **Max chunk reduced to 10 seconds**: Most in-game callouts are 1-3 seconds. 10s is plenty while keeping transcription responsive.
- **Why it worked on macOS**: Different mic with cleaner signal. The threshold should ideally be calibrated per-mic (adaptive thresholding), or use a neural VAD like Silero.

### Multilingual STT

- **ggml-small.bin vs ggml-small.en.bin**: Same size (466 MB), but multilingual supports 99 languages. English WER is ~4.4% vs ~3.4% — negligible difference for game chat.
- **Per-client locale**: Each audio message includes a locale prefix (`[1-byte length][locale string]`). The server is stateless — no per-client session tracking for STT. The locale tells whisper what language the audio is in, not what language to output.
- **Auto-detect vs specified**: `language=auto` adds ~1 second overhead for language detection. In production, the player's language comes from their profile — faster and more accurate.

### Broadcast Architecture

- **Client registry**: Thread-safe `std::map<int, ClientInfo>` mapping client_id → socket + locale. Added on connect, removed on disconnect.
- **Broadcast model**: When a client speaks, the transcription is sent back to the sender AND broadcast to all other connected clients.
- **Listen-only mode**: Test client accepts `--listen` flag — connects, registers locale, receives broadcasts without capturing mic. Useful for testing broadcast with one mic.
- **Zero-length audio = registration**: A message with locale + zero audio length registers the client in the broadcast list without triggering transcription.
- **Why stateless locale**: Each audio message carries its own locale so the server doesn't need to track per-client language state. The client registry exists for broadcast routing and future translation, not for STT.

### Translation Architecture (Planned, Not Implemented)

- **STT and translation are completely separate problems**: STT = audio → text (same language). Translation = text → text (different language). They share nothing technically.
- **NLLB-200 (Meta)**: Single 1.2 GB model, 200 languages, any-to-any translation. ~15-30ms per sentence on GPU. CC-BY-NC license (free for prototype, needs commercial license for production).
- **Graph theory**: Every player is a node with a language attribute. Every communication is an edge that needs a translation transform. The server routes text through the right NLLB-200 translation based on sender and receiver locales.
- **Production architecture**: STT runs on the player's local machine (fast, no round trip). Only text is sent to the server. Server handles translation only — much less GPU load.
- **Console constraints**: Can't ship 1.2 GB models to consoles. Translation must be server-side. STT could also be server-side for console clients.

### Unreal Engine Integration

- **6 files, zero game-specific code**: WhisperClient (TCP), VoiceChatComponent (mic + push-to-talk), VoiceChatWidget (chat overlay). Drop into any Unreal project.
- **Push-to-talk**: Ctrl+V checked every frame in TickComponent. StartRecording on press, StopRecording on release.
- **Audio format conversion**: Captured audio (48kHz, multi-channel, float) → 16kHz mono 16-bit PCM via channel averaging + linear interpolation resampling.
- **Background receive loop**: WhisperClient runs a continuous receive thread so it can receive broadcasts even when not recording. Previously it was request-response (send audio → block waiting for response), which meant it missed broadcasts while idle.
- **Registration on connect**: Sends locale + zero-length audio immediately on connect so the server registers it for broadcasts before anyone presses Ctrl+V.
